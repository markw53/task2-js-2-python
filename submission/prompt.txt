# Task: Translate mdforge from JavaScript to Python

## Source Repository

The directory `/source` contains **mdforge**, a command-line toolkit for parsing, validating, and transforming Markdown files. It is written in JavaScript (Node.js) and provides the following CLI subcommands:

- `parse <file>` — Parse a Markdown file and output structured metadata as JSON (frontmatter, headings, links, code blocks, word count).
- `toc <file>` — Generate a Markdown-formatted table of contents from a file's headings.
- `html <file>` — Convert a Markdown file to simplified HTML.
- `stats <path>` — Compute word/line/heading/link/code-block statistics for a single file or an entire directory of Markdown files, output as JSON.
- `validate <dir>` — Validate all internal and anchor links across a directory of Markdown files. Exits with code 2 if any broken links are found.
- `batch <dir> -o <outdir>` — Batch-process a directory of Markdown files, writing per-file JSON metadata to an output directory.

The source repo contains approximately 15 JavaScript source files covering: a CLI entry point (using Commander.js), a Markdown parser (frontmatter extraction, heading/link/code-block extraction, word counting), HTML conversion, TOC generation, statistics computation, link validation, file I/O utilities, logging, error handling, and configuration.

## Your Task

Translate the entire mdforge repository into idiomatic **Python 3.11**. Place all translated code in `/target`. The result must be a fully functional, installable Python package that is behaviorally equivalent to the JavaScript source.

## Requirements

### Package Structure
- The Python package must be named `mdforge`.
- It must be installable via `pip install -e .` (i.e., provide a `setup.py` or `pyproject.toml` with appropriate metadata).
- The package must expose a CLI entry point so that after installation, `python -m mdforge` invokes the CLI.

### CLI Interface
- Use the **Click** library for CLI argument parsing.
- The CLI must support the same six subcommands listed above with identical argument and option semantics:
  - `mdforge parse <file>`
  - `mdforge toc <file>`
  - `mdforge html <file>`
  - `mdforge stats <path>`
  - `mdforge validate <dir>`
  - `mdforge batch <dir> -o <outdir>`
- All JSON output must be written to **stdout**. Log/status messages must be written to **stderr**.

### Behavioral Equivalence
- **Parse output**: JSON output from `parse` must include the keys `file`, `frontmatter`, `headings`, `links`, `codeBlocks`, `wordCount`. Heading objects must include `level`, `text`, `slug`, `line`. Link objects must include `text`, `url`, `type`, `line`. Code block objects must include `language`, `code`, `startLine`, `endLine`.
- **TOC generation**: Must produce Markdown-formatted TOC lines. H1 headings are excluded. Indentation is 2 spaces per heading level below H2. Each entry links to its slug anchor (e.g., `- [Heading](#slug)`).
- **HTML conversion**: Must handle headings, bold, italic, inline code, fenced code blocks (with language class), links, unordered lists, ordered lists, blockquotes, horizontal rules, and paragraphs. Must strip YAML frontmatter before conversion. Fenced code blocks emit `<pre><code class="language-{lang}">` when a language is specified.
- **Statistics**: Single-file stats must include `file`, `wordCount`, `lineCount`, `headingCount`, `linkCount`, `codeBlockCount`, `internalLinkCount`, `externalLinkCount`, `anchorLinkCount`, `hasTitle`, `hasFrontmatter`, `codeLanguages`. Directory stats must aggregate across all `.md` files and include `fileCount`, `totalWordCount`, `totalLineCount`, `totalHeadings`, `totalLinks`, `totalCodeBlocks`, `averageWordCount` (rounded to nearest integer), `files`, `codeLanguages`.
- **Link validation**: Must validate internal file links and anchor links. External links and `mailto:`/`ftp://` links are ignored. Broken links must be reported with `source`, `line`, `text`, `url`, `type`, `reason`. The command must exit with code 2 when broken links are found.
- **Batch processing**: Must write one JSON file per input Markdown file to the output directory. JSON must include `file`, `metadata`, `headings`, `links`, `codeBlocks`, `stats`, `toc`. Must report `processed` and `failed` counts in stdout JSON output.

### Frontmatter Handling
- Use the **python-frontmatter** library for YAML frontmatter parsing.
- Malformed frontmatter must not crash the parser — treat the entire content as body with empty frontmatter data.

### Slug Generation
- Slugs are generated by: lowercasing, removing non-alphanumeric characters (except hyphens and word characters), replacing whitespace with hyphens, collapsing consecutive hyphens, and stripping leading/trailing hyphens.

### Code Block Handling
- Fenced code blocks (both backtick and tilde fences) must be detected. Headings and links inside code blocks must be ignored during extraction. Unclosed code blocks are discarded (not included in output).

### Word Counting
- Word count must exclude content inside fenced code blocks and inline code spans.

### Exit Codes
- Exit code 0 on success.
- Exit code 1 on errors (file not found, empty directory, I/O errors).
- Exit code 2 specifically for link validation failure (broken links found).

### Error Handling
- Use a custom exception class that carries an exit code.
- File-not-found and I/O errors must produce meaningful error messages to stderr and exit with the appropriate code.

### File Discovery
- Directory scanning must recursively find all `*.md` files. Results must be sorted for deterministic output.

## What NOT to Translate
- Do not translate any test files.
- Do not translate Node.js-specific dev tooling (eslint configs, .prettierrc, etc.).
- Do not include the JavaScript source code in `/target`.

## Build and Verification
- `cd /target && pip install -e .` must succeed.
- After installation, `python -m mdforge parse <file>` (and all other subcommands) must work.
